# 정산 이후 소비 관련 프론트 액션 제한 구현 계획

- GitHub Issue: https://github.com/together-trip/togethertrip-client-flutter/issues/23
- 작업 브랜치: `fix/issue-23-settlement-expense-ui-lock`
- 기준 브랜치: `develop`
- 작성일: 2026-06-19

## 배경

정산이 확정된 여행에서는 소비 생성/수정/삭제가 막혀야 한다. 현재 백엔드는 거래 생성/삭제를 차단하지만, 앱의 여행 상세 화면은 정산 완료 상태에서도 소비 등록 선택지와 소비 게시글 수정/삭제 액션을 그대로 보여준다.

그 결과 사용자는 소비 폼을 끝까지 작성한 뒤 서버 에러를 받거나, 소비 게시글을 수정/삭제할 수 있다고 오해할 수 있다.

정책 목표는 다음과 같다.

- 정산 이후에도 일반 기록 작성/수정/삭제는 허용한다.
- 정산 이후 소비 등록 진입은 막는다.
- 정산 이후 소비 게시글 수정/삭제 액션은 숨기거나 비활성화한다.
- 서버 에러에만 의존하지 않고 화면에서 정책을 명확히 표현한다.

## TogetherTrip 에이전트 판단

### Planner

- 이번 이슈는 client-flutter의 UX 정책 정합성 보강이다.
- 백엔드 정책 변경은 server-main #80에서 처리한다.
- 사용자가 폼을 다 작성한 뒤 실패하는 흐름을 줄이는 것이 핵심이다.

### UI Controller

- `+` 버튼은 유지하되, 정산 이후 생성 선택 바텀시트에서 `기록`만 바로 실행 가능하게 한다.
- `소비`는 숨김 또는 disabled 상태 중 하나로 통일한다.
- 추천은 disabled 상태와 짧은 안내 문구다. 사용자가 왜 소비를 못 쓰는지 이해할 수 있다.

### TDD Guide

- `TripDetailScreen` 위젯 테스트를 추가하거나 기존 테스트에 정산 상태별 케이스를 보강한다.
- `NOT_STARTED`에서는 기록/소비 모두 가능해야 한다.
- `SETTLED`에서는 기록만 가능하고 소비는 폼으로 진입하지 않아야 한다.

### Verify Agent

- `flutter test`를 실행한다.
- UI 변경이 작은 경우에도 좁은 화면에서 버튼 텍스트가 깨지지 않는지 확인한다.

## 범위

- `TripDetailScreen._openCreateChooser()`에서 정산 상태를 기준으로 소비 선택 가능 여부를 판단한다.
- `settlementStatus != 'NOT_STARTED'`이면 소비 등록 폼으로 진입하지 않는다.
- 정산 이후 `EXPENSE` 게시글의 수정/삭제 액션을 제한한다.
- 정산 이후 `RECORD` 게시글의 수정/삭제 액션은 유지한다.
- 정산 잠금 관련 서버 에러가 발생하면 사용자 친화적인 메시지로 표시한다.

## 제외 범위

- 백엔드 소비 게시글 수정 정책 변경
- 정산 확정 취소 또는 재정산 UX
- 소비 금액/참여자 수정 화면 신규 구현
- 환율 화면의 전역 환율 조회 정책 변경

## 설계

### 생성 선택 바텀시트

현재 흐름:

1. `FloatingActionButton` 클릭
2. `_openCreateChooser()`
3. `기록` 또는 `소비` 선택
4. 소비 선택 시 `_openExpenseForm()`

변경안:

- `final isExpenseLocked = _trip?.settlementStatus != 'NOT_STARTED'`
- `기록` 선택지는 항상 활성화한다.
- `소비` 선택지는 `isExpenseLocked`이면 disabled 처리한다.
- disabled 소비 선택지 아래 또는 tile subtitle로 `정산 완료 후에는 소비를 추가할 수 없어요.`를 표시한다.

### 게시글 액션 바텀시트

현재 작성자 게시글이면 수정/삭제 액션을 항상 보여준다.

변경안:

- `final isLockedExpensePost = post.postType == 'EXPENSE' && _trip?.settlementStatus != 'NOT_STARTED'`
- `isLockedExpensePost`이면 수정/삭제 액션을 숨기거나 disabled 처리한다.
- 일반 기록은 기존 액션을 유지한다.
- disabled를 선택하면 안내 문구 `정산 완료 후에는 소비 기록을 변경할 수 없어요.`를 함께 표시한다.

### 에러 메시지

서버가 정산 잠금 에러를 반환하는 경우 화면에는 다음처럼 정리된 메시지를 보여준다.

- 소비 등록 실패: `정산 완료 후에는 소비를 추가할 수 없어요.`
- 소비 수정/삭제 실패: `정산 완료 후에는 소비 기록을 변경할 수 없어요.`

## 테스트 계획

### 위젯 테스트

- `NOT_STARTED` 여행에서 `+` 선택 바텀시트에 기록/소비가 모두 활성화된다.
- `SETTLED` 여행에서 `+` 선택 바텀시트의 기록은 활성화되고 소비는 비활성화된다.
- `SETTLED + RECORD` 게시글은 수정/삭제 액션을 볼 수 있다.
- `SETTLED + EXPENSE` 게시글은 수정/삭제 액션이 제한된다.

### 수동 확인

- 좁은 화면에서 안내 문구와 액션 tile 텍스트가 겹치지 않는다.
- 정산 상태 배지가 기존처럼 표시된다.
- 정산 미시작 여행의 기존 소비 등록 플로우가 깨지지 않는다.

## 위험과 확인 사항

- 현재 `TripDetailScreen` 테스트가 없다면 테스트 fixture 작성 비용이 있을 수 있다.
- 소비 게시글 수정 전체 차단과 일부 메타데이터 허용 정책은 server-main #80과 맞춰야 한다.
- 액션을 숨길지 disabled로 둘지는 사용자 이해도를 기준으로 결정한다. 추천은 disabled + 짧은 안내다.

## 완료 기준

- 정산 완료 이후 앱에서 소비 등록 폼으로 진입할 수 없다.
- 정산 완료 이후 소비 게시글 수정/삭제 가능하다는 잘못된 UX가 사라진다.
- 정산 완료 이후에도 일반 기록 작성/수정/삭제는 가능하다.
- `flutter test`가 통과한다.
