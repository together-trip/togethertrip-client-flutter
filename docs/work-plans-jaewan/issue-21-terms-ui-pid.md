# Issue 21 약관 플로우와 UI PID 제어 개선 사례

- Client Issue: https://github.com/together-trip/togethertrip-client-flutter/issues/21
- Server Follow-up: https://github.com/together-trip/togethertrip-server-main/issues/74
- 대상 앱: TogetherTrip Flutter app
- 핵심 키워드: 약관 동의 플로우, UI 일관성, 비용 함수, PID식 반복 보정, 공통 컴포넌트화

## 한 줄 요약

약관 화면을 단순 추가하는 데서 끝내지 않고, 여행/환율 화면을 기준점으로 삼아 전체 앱 UI 오차를 수치화하고 PID 제어 아이디어로 반복 보정했다.
그 결과 회원가입 약관 플로우, 마이페이지 약관 조회, 공통 디자인 토큰, 바텀시트/버튼/에러 패턴을 정리하면서도 `flutter analyze`와 `flutter test 62개`를 통과했다.

## 문제 배경

Issue 21의 표면적인 요구는 약관 동의 화면과 UI 통일성 개선이었다.
하지만 실제로는 세 가지 문제가 섞여 있었다.

1. 약관 동의 시점이 불명확했다.
   온보딩 첫 화면에 약관을 넣을지, 카카오 로그인 이후 신규 회원가입 시점에 넣을지 결정이 필요했다.

2. 마이페이지의 `약관` 메뉴가 준비 중 상태였다.
   가입 후 사용자가 약관을 다시 확인할 수 있는 읽기 화면이 필요했다.

3. 새 화면을 추가하면 기존 여행/환율 UI와 어긋날 가능성이 있었다.
   화면마다 색상, radius, 버튼, 바텀시트, 에러 텍스트를 직접 선언하면 앱 전체 일관성이 계속 흔들릴 수 있었다.

초기에는 "약관 화면을 만들자"로 시작할 수 있었지만, 최종 목표는 "앱 전체 기준에서 UI 오차를 줄이는 것"으로 확장했다.

## 결과물

### 기능

- 카카오 로그인 이후 신규 가입자에게만 약관 동의 화면 노출
- 기존 회원은 약관 화면 없이 메인으로 이동
- 약관 동의 저장은 `TermsAgreementService` mock 경계로 분리
- 마이페이지 `약관` 메뉴에서 약관 목록 화면으로 이동
- 약관 상세는 바텀시트로 확인
- 백엔드 저장/조회 API는 server issue #74로 분리

### 포트폴리오 관점에서 보강한 부분

처음 기록은 루프마다 수치를 이어 붙인 작업 로그에 가까웠다.
그 방식은 진행 과정은 남지만, 읽는 사람이 "왜 이 판단이 좋았는지"를 따라가기 어렵다.

그래서 최종 문서는 다음 세 가지가 드러나도록 다시 정리했다.

- 단순 요청 처리, 단순 루프, PID 루프가 각각 어떻게 다른지
- PID 제어가 원래 무엇이고, 이번 작업에서는 어디까지 차용했는지
- 수치가 단순 장식이 아니라 의사결정과 멈춤 판단에 어떻게 쓰였는지

즉, 이 문서는 "무엇을 만들었는가"뿐 아니라 "왜 그 방식으로 개선했는가"를 설명하는 사례 문서로 정리했다.

피드백 이후 메인 환율, 계산 결과, 히스토리 금액, 섹션 제목을 `w800`으로 낮춰 강조는 유지하되 과한 굵기는 줄였다.

## PID 제어란 무엇인가

PID 제어는 제어공학에서 목표값과 현재값의 차이를 줄이기 위해 쓰는 피드백 제어 방식이다.
예를 들어 온도 조절기를 생각하면, 목표 온도와 현재 온도의 차이를 보고 난방 출력을 계속 조절한다.

핵심은 한 번에 정답을 맞히는 것이 아니라, 현재 오차를 계속 관측하면서 보정량을 조절하는 것이다.

```text
목표값 setpoint: 도달하고 싶은 상태
현재값 process value: 지금 시스템의 상태
오차 error: 목표값 - 현재값
제어 입력 control input: 오차를 줄이기 위해 가하는 조정
피드백 feedback: 조정 후 다시 측정한 결과
```

PID는 세 가지 항으로 구성된다.

### P: Proportional

현재 오차에 비례해서 바로 보정한다.
목표와 현재 차이가 크면 강하게 조정하고, 차이가 작으면 작게 조정한다.

UI 작업에 대입하면:

- 지금 화면에서 가장 튀는 문제를 먼저 고친다.
- 예: 0 radius 버튼이 기준 화면의 8px radius와 다르면 바로 8px로 맞춘다.

### I: Integral

오차가 반복해서 누적되는지를 본다.
한 번의 문제가 아니라 여러 화면에서 계속 반복되면, 개별 수정이 아니라 구조적으로 해결한다.

UI 작업에 대입하면:

- 같은 색상, radius, 바텀시트, 에러 스타일이 여러 화면에 반복되면 공통 컴포넌트로 뺀다.
- 예: `showModalBottomSheet` 직접 호출이 여러 화면에 있으면 `showAppBottomSheet`로 기준을 고정한다.

### D: Derivative

변화 추세를 본다.
보정할수록 오히려 복잡도나 부작용이 커지는지 감지한다.

UI 작업에 대입하면:

- 공통화를 더 하면 코드가 너무 추상화되는지 본다.
- 화면 전환 수가 늘거나 구현 복잡도가 커지면 멈춘다.
- 예: 환율 화면의 `w900`은 처음에는 의도적 강조로 보였지만, 실제 피드백 이후 과한 시각 무게로 판단해 `w800`으로 낮췄다.

## 이 프로젝트에서 PID를 어떻게 적용했나

일반적인 PID 수식을 그대로 코드에 넣은 것은 아니다.
대신 PID의 사고방식을 UI 개선 루프에 맞게 해석했다.
따라서 이 문서에서의 PID는 실제 런타임 제어기라기보다, 디자인/구현 의사결정을 피드백 기반으로 반복 보정하기 위한 작업 모델이다.

```text
목표값: 여행/환율 화면의 UI 컨벤션
현재값: 각 화면의 실제 UI 상태
오차: 기준 화면과의 차이
제어 입력: 화면 수정, 공통 컴포넌트 추출, 스타일 보정
피드백: flutter test, flutter analyze, 코드 검색, 비용 점수 변화
```

이때 오차는 다음처럼 봤다.

```text
e_before = 보정 전 기준 화면과의 차이
e_after = 보정 후 기준 화면과의 차이
delta = e_after - e_before

delta < 0 이면 개선
delta = 0 이면 시각 변화 없음 또는 유지
delta > 0 이면 보정으로 새 비용 발생
```

단순 루프에서는 보통 "테스트가 통과했는가"가 종료 조건이 된다.
PID 루프에서는 테스트 통과뿐 아니라 `delta`가 충분히 작아졌는지, 반복 오차가 남았는지, 보정으로 복잡도가 커졌는지를 함께 본다.
이 차이 때문에 같은 테스트 통과 상태라도, PID 루프에서는 한 번 더 "아직 남은 UI 오차가 의미 있는가"를 확인한다.

기준 화면은 현재 앱에서 가장 안정적인 여행/환율 화면으로 잡았다.

- 흰 배경
- 검정 중심 색상 `0xFF1A1A1A`
- 보조 텍스트 `0xFF6B6B6B`
- 위험 색상 `0xFFCC0000`
- 주요 CTA 높이 52, 검정 배경, 흰 글자
- 카드/버튼/입력 radius 8
- 바텀시트 top radius 8, 36x4 핸들
- 정보 밀도는 높게, 설명은 짧게

## 비용 함수

각 항목은 0~5점으로 평가했다.
점수가 낮을수록 기준 화면과 가깝다.

```text
가중 비용 =
  입력 단계 수 * 1.3
+ 선택해야 하는 항목 수 * 1.1
+ 헷갈릴 가능성 * 1.6
+ 한 손 조작 어려움 * 1.2
+ 정산 개념 노출 정도 * 1.4
+ 화면 전환 수 * 1.2
+ 구현 복잡도 * 1.0
+ 기존 화면과의 불일치 * 0.9
```

이 비용 함수는 절대적인 정답을 만들기 위한 점수가 아니다.
루프마다 어떤 오차가 줄었는지, 그리고 보정으로 복잡도가 얼마나 늘었는지 비교하기 위한 계기판이다.

## 단순 요청 vs 단순 루프 vs PID 루프

이번 작업의 포트폴리오 포인트는 이 3단 비교다.

| 항목 | 단순 요청 처리 | 단순 루프 엔지니어링 | PID 루프 엔지니어링 |
| --- | --- | --- | --- |
| 시작점 | 사용자가 말한 화면을 바로 수정 | 계획, 구현, 검증 순서로 진행 | 기준 화면과 목표 오차를 먼저 정의 |
| 판단 기준 | 요청한 기능이 동작하는가 | 테스트가 통과하는가 | 기준 대비 오차가 얼마나 줄었는가 |
| 피드백 | 사용자 재요청 | 테스트 실패, 리뷰 | 비용 변화량, 누적 오차, D 경고 |
| 공통화 | 필요해 보이면 수동 정리 | 중복 발견 후 추출 | 반복 오차가 쌓이면 I 보정으로 추출 |
| 과수정 방지 | 사용자가 멈춰야 함 | 계획 범위로 제어 | 복잡도/화면 전환 증가를 D 경고로 제한 |
| 산출물 | 구현 코드 | 구현 코드 + 검증 결과 | 구현 코드 + 검증 결과 + 오차 감소 로그 |
| 설명력 | "화면 구현" | "계획-구현-검증 완료" | "오차를 제어 방식으로 낮춘 개선 사례" |

### 같은 요구를 처리했을 때의 차이

요구:

```text
이슈 21에 맞는 구현 범위와 첫 온보딩 페이지 디자인이 필요하다.
약관은 카카오 로그인 이후 회원가입 시점이 좋다.
마이페이지에도 약관을 만들어야 한다.
전체 화면 통일성을 맞추고 싶다.
```

단순 요청 처리라면:

- 약관 화면을 만들고 라우팅을 연결한다.
- 마이페이지 약관 메뉴를 연결한다.
- 화면이 동작하면 끝난다.

단순 루프 엔지니어링이라면:

- 관련 파일을 찾고 계획을 세운다.
- 구현 후 `flutter analyze`, `flutter test`를 돌린다.
- 실패하면 수정한다.
- 통과하면 작업을 닫는다.

PID 루프 엔지니어링에서는:

- 약관 위치가 온보딩인지 카카오 이후인지 먼저 흐름 비용으로 판단한다.
- 여행/환율 화면을 기준점으로 잡고 새 화면이 얼마나 어긋나는지 계산한다.
- 즉시 큰 오차는 P 보정으로 줄인다.
- 반복 스타일 오차는 I 보정으로 공통화한다.
- 공통화가 과해지는 순간은 D 경고로 멈춘다.
- 테스트 실패를 단순 실패가 아니라 새로운 비용 신호로 기록한다.

즉, 단순 루프는 "작업을 안전하게 끝내는 방식"에 가깝고, PID 루프는 "작업을 끝내면서도 기준과의 차이를 계속 줄이는 방식"에 가깝다.
이 차이가 포트폴리오에서 중요하다.
단순 루프는 성실한 구현 과정을 보여주지만, PID 루프는 사용성/일관성/복잡도를 동시에 제어한 판단 과정을 보여준다.

정리하면, 단순 요청은 결과 중심이고, 단순 루프는 절차 중심이며, PID 루프는 기준과 오차 중심이다.
이번 문서의 핵심은 바로 이 세 방식의 차이를 구현 사례로 보여주는 데 있다.

## 핵심 의사결정

### 약관 동의는 카카오 로그인 이후로 배치

온보딩 첫 화면에 약관을 넣으면 사용자가 기존 회원인지 신규 회원인지 모르는 상태에서 동의를 요구하게 된다.
카카오 로그인 이후에는 서버 응답으로 `AUTHENTICATED`, `PROFILE_REQUIRED`, `PHONE_VERIFICATION_REQUIRED` 상태가 나뉘므로, 신규 가입자에게만 약관 동의를 요구할 수 있다.

최종 흐름:

```text
온보딩
-> 카카오 로그인
-> 기존 회원이면 메인
-> 신규 회원이면 약관 동의
-> 프로필 설정 또는 전화번호 인증
```

이 결정으로 불필요한 사전 동의 비용을 줄이고, 회원가입 의사가 확인된 뒤에만 필수 약관 동의를 요구하도록 만들었다.

### 백엔드 API는 mock service 경계로 분리

약관 저장/조회 API가 아직 없었기 때문에 화면 내부 임시 state로 처리하지 않고 `TermsAgreementService`를 만들었다.
이 service는 현재 mock 데이터를 반환하지만, 서버 API가 준비되면 교체할 수 있는 경계다.

서버 follow-up:

- https://github.com/together-trip/togethertrip-server-main/issues/74

이 결정으로 프론트 구현은 진행하면서도 API 미정 상태가 화면 코드에 퍼지는 것을 막았다.

### 마이페이지 약관은 "동의"가 아니라 "조회"로 분리

회원가입 약관 화면은 필수 동의가 목적이다.
반면 마이페이지 약관은 가입 후 다시 확인하는 읽기 화면이다.

그래서 같은 약관 데이터를 쓰되 역할을 분리했다.

- `TermsAgreementScreen`: 신규 가입자 필수 동의
- `TermsListScreen`: 마이페이지 약관 조회

## PID가 드러난 구체 사례

### P 보정: 현재 오차를 즉시 낮춘 사례

여행 생성 화면의 0 radius 버튼/입력은 기준 화면과 가장 눈에 띄게 달랐다.

```text
기존 화면과의 불일치: 3 -> 1
구현 복잡도: 1 -> 1
가중 비용: 14.3 -> 12.5
감소량: -1.8
```

이 보정은 구조를 바꾸지 않고 현재 오차만 줄인 전형적인 P 보정이다.

### I 보정: 반복 오차를 공통화한 사례

색상, radius, 에러 텍스트, 바텀시트 선언이 여러 화면에 반복됐다.

```text
반복 스타일 선언 비용: 4 -> 2
기존 화면과의 불일치 재발 위험: 3 -> 2
구현 복잡도: 2 -> 3
가중 비용: 12.5 -> 11.2
감소량: -1.3
```

반복 오차는 줄었지만 구현 복잡도는 올라갔다.
그래서 모든 화면을 한 번에 대공사하지 않고 최근 수정 화면, 기준 화면, 관련 화면 순서로 나눠 적용했다.

이후 전체 화면을 같은 기준으로 다시 훑으면서, 특정 화면을 기준으로 삼는 방식에서 공통 UI 토큰 자체를 기준으로 삼는 방식으로 전환했다.

```text
feature 내부 직접 핵심 색상 사용: 여러 화면 -> 0곳
feature 내부 일반 버튼 styleFrom: 여러 화면 -> 0곳
showModalBottomSheet 직접 사용: core 내부 1곳만 유지
w900 직접 사용: 0곳
불필요한 초록 계열 상태 색상: 여러 곳 -> 0곳
온보딩 페이지 전환: dots-only -> PageView 3장
하단 구분선/Divider: 주요 화면 여러 곳 -> 브랜드 마크/의도적 구조 요소만 유지
여행 목록 카드 테두리: 검정 테두리 -> soft line
```

이 루프는 I 보정의 확장이다.
반복 오차가 눈에 띄는 화면에만 남아 있는지 보지 않고, 모든 화면을 같은 규칙으로 다시 검사해 재발 경로를 줄였다.
다만 여행 상세의 기록/소비 피드는 일반 폼 컨트롤과 다르게 취급했다.
사용자가 기대한 "인스타 피드 같은 느낌"은 기능 정체성에 가까우므로, borderless 기준을 피드 카드 자체에 과하게 적용하지 않았다.

### D 경고: 기준 판단을 피드백으로 갱신한 사례

환율 화면에 남은 `fontWeight.w900`은 처음에는 제거하지 않았다.
검색상으로는 예외처럼 보였지만, 당시에는 환율 화면이 기준 화면이고 w900은 금액/핵심 수치 강조에 쓰인다고 판단했다.

하지만 다시 화면 무게를 평가했을 때, w900이 메인 환율, 계산 결과, 히스토리 금액, 제목에 반복되어 화면 전체가 과하게 굵어졌다.
이 피드백을 반영해 환율 화면의 남은 w900도 w800으로 낮췄다.

```text
초기 판단: 오차가 아니라 기준 신호
재평가: 강조가 여러 곳에 누적되어 시각 무게 증가
결정: w900 -> w800
```

이것이 D 경고의 역할이다.
모든 차이를 무조건 제거하는 것도 아니고, 한 번 기준으로 본 판단을 고정하는 것도 아니다.
피드백 이후 실제 화면 비용이 커졌다면 기준 판단도 갱신한다.

### 테스트 실패를 제어 신호로 사용한 사례

온보딩 placeholder를 카드형 비주얼로 바꾼 뒤, 테스트 기본 viewport에서 overflow가 발생했다.

```text
1차 overflow: 45px
2차 overflow: 1px
최종 overflow: 0px
```

단순 루프도 테스트 실패를 고칠 수는 있다.
하지만 PID 루프에서는 이를 비용으로 다시 기록했다.

```text
작은 viewport overflow 비용: 5 -> 0
```

즉, 테스트 실패가 "고쳐야 할 버그"에서 끝난 것이 아니라 "새로 생긴 UI 비용"으로 다시 모델에 들어갔다.

## 면접에서 설명할 수 있는 포인트

이 작업을 설명할 때는 "PID를 실제 알고리즘으로 구현했다"라고 말하면 과장이다.
정확한 표현은 "PID 제어의 사고방식을 UI 개선 루프에 적용했다"이다.

말할 수 있는 요지:

```text
기준 화면을 setpoint로 두고, 각 화면의 스타일 차이를 error로 봤습니다.
큰 현재 오차는 P 보정으로 바로 낮췄고, 여러 화면에 반복되는 오차는 I 보정으로 공통 컴포넌트화했습니다.
반대로 공통화가 과해져 구현 복잡도가 올라가거나 기준 화면의 의도까지 지울 위험이 있으면 D 경고로 멈췄습니다.
```

구체 예시:

- P: 여행 생성 화면의 0 radius를 8px로 맞춰 불일치 비용 `3 -> 1`
- I: 반복되는 바텀시트/에러/버튼 스타일을 `app_design.dart`로 공통화
- D: 환율 화면의 `w900`은 재평가 후 시각 무게가 과하다고 보고 `w800`으로 낮춤
- Feedback: 테스트가 잡은 온보딩 overflow `45px -> 1px -> 0px`

이 설명은 "감으로 UI를 고쳤다"가 아니라 "기준과 오차를 정의하고, 보정 결과를 측정했다"는 점을 보여준다.

## 포트폴리오용 설명

TogetherTrip 앱의 약관 동의 플로우와 UI 통일성을 개선하면서, 단순 구현 루프를 PID 제어 아이디어로 확장했다.
먼저 여행/환율 화면을 기준점으로 정의하고, 화면 간 오차를 비용 함수로 수치화했다.
이후 큰 현재 오차는 P 보정으로 즉시 낮추고, 반복되는 스타일 오차는 I 보정으로 공통 컴포넌트화했으며, 복잡도나 과도한 통일화가 커지는 지점은 D 경고로 제한했다.

이 접근 덕분에 약관 동의 위치를 카카오 로그인 이후 신규 가입 시점으로 자연스럽게 배치했고, 백엔드 API가 없는 상태에서도 mock service 경계를 만들어 추후 연동 비용을 낮췄다.
또한 색상, radius, 에러 텍스트, 바텀시트, 주요 버튼 패턴을 공통화해 앱 전체의 UI 재발 오차를 줄였다.
중간에 발생한 45px overflow는 테스트 신호를 비용으로 다시 반영해 0px까지 줄였고, 최종적으로 `flutter analyze`와 `flutter test 62개`를 통과했다.

이 사례의 핵심은 "화면을 예쁘게 고쳤다"가 아니라, "기준 화면 대비 오차를 측정하고, 제어 루프로 반복 보정하며, 과보정은 중단했다"는 점이다.
